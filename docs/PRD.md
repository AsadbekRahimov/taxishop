# TaxiShop — Product Requirements Document (PRD)

## 1. Обзор продукта

**TaxiShop** — e-commerce платформа для продажи товаров пассажирам такси. Планшет установлен на заднем сиденье автомобиля — клиент (пассажир) сам просматривает каталог и оформляет заказы. Водитель подтверждает заказы со своего смартфона через отдельный кабинет. Администратор управляет каталогом и заказами через панель Filament.

**Три интерфейса:**
- **Магазин (планшет, заднее сиденье)** — клиент просматривает товары и оформляет заказы
- **Кабинет водителя (смартфон водителя, `/driver/orders`)** — водитель подтверждает/отклоняет заказы
- **Админ-панель (`/admin`)** — управление каталогом, водителями, заказами

**Целевая аудитория:**
- Пассажиры такси (клиенты — используют планшет в машине)
- Водители такси (подтверждают заказы с телефона)
- Администраторы (управление через админ-панель)

## 2. Текущий функционал

### 2.1 Аутентификация и роли

| Функция | Описание |
|---------|----------|
| Вход по логину | Водители и администраторы входят через поле `login` (не email) |
| Роли | `driver` — водитель, `admin` — администратор |
| Флаг активности | Неактивные пользователи (`is_active = false`) не могут войти |
| Remember me | Поддержка «запомнить меня» |
| Throttle | Ограничение 5 попыток входа в минуту |
| Планшет и телефон | Оба устройства авторизованы под одним аккаунтом водителя |

### 2.2 Каталог товаров

| Функция | Описание |
|---------|----------|
| Категории | Иерархические (parent → children), с иконками и сортировкой |
| Товары | Название, описание, цена, старая цена (зачёркнутая), главное изображение, доп. изображения |
| Slug-маршруты | Каждый товар и категория имеют уникальный slug для ЧПУ |
| Наличие у водителя | Отображается количество товара в машине текущего водителя |
| Пагинация | 12 товаров на страницу в категории и поиске |
| Сортировка | По популярности (кол-во заказов), цене ↑/↓, новизне |

### 2.3 Поиск

| Функция | Описание |
|---------|----------|
| Текстовый поиск | По названию товара (LIKE, регистронезависимый) |
| Пагинация | 12 результатов на страницу |
| Наличие | Показывается количество в стоке водителя |

### 2.4 Корзина

| Функция | Описание |
|---------|----------|
| Хранение | В сессии (не в БД) |
| Добавление | Указание товара и количества |
| Обновление | Изменение количества; при qty ≤ 0 товар удаляется |
| Удаление | Удаление отдельного товара |
| Подсчёт | Итоговая сумма и количество позиций |
| Бейдж | Количество товаров отображается в шапке сайта |

### 2.5 Типы заказов

Система поддерживает два типа заказов:

| Тип | Описание | Данные клиента |
|-----|----------|----------------|
| **Pickup (на месте)** | Клиент покупает товар прямо в машине у водителя | Не требуются (имя, телефон необязательны) |
| **Delivery (доставка)** | Клиент заказывает доставку на дом | Имя, телефон (+998), адрес доставки обязательны |

### 2.6 Оформление заказа (Checkout)

| Функция | Описание |
|---------|----------|
| Выбор типа заказа | Переключение между «Забрать на месте» и «Доставка на дом» |
| Данные клиента (только для доставки) | Имя, телефон (формат +998), адрес доставки |
| Способы оплаты (для обоих типов) | `cash` — наличные, `qr` — QR-код, `delivery` — при получении/доставке |
| Номер заказа | Формат `TS-XXXXXX` (авто-генерация) |
| Статус при создании | Всегда `pending` — ожидает подтверждения водителем |
| Страница ожидания | После оформления показывается «Ожидание подтверждения водителя» |

### 2.7 Подтверждение водителем (Кабинет водителя)

| Функция | Описание |
|---------|----------|
| Отдельный интерфейс | `/driver/orders` — мобильная страница для смартфона водителя |
| Отдельный layout | Свой дизайн (зелёная шапка, без магазинных элементов) |
| Ожидающие заказы | Список заказов со статусом `pending` |
| Подтверждение | Водитель нажимает «Подтвердить» → статус `confirmed`, сток списывается (для pickup) |
| Отклонение | Водитель нажимает «Отклонить» → статус `cancelled` |
| История | Вкладка с подтверждёнными/отклонёнными заказами |
| Детали заказа | Тип, клиент, адрес, способ оплаты, состав, сумма |

**Важно:** Клиент НЕ видит кабинет водителя. Планшет (магазин) и телефон (кабинет) — полностью раздельные интерфейсы.

### 2.8 Управление стоком водителя

| Функция | Описание |
|---------|----------|
| Таблица `driver_stock` | Связь водитель ↔ товар с количеством |
| Уникальность | Один товар — одна запись на водителя |
| Декремент | Сток списывается ТОЛЬКО после подтверждения водителем (для pickup) |
| Удаление записи | Если количество достигает 0 — запись удаляется |
| Для delivery | Сток НЕ списывается (товар доставляется со склада) |

### 2.9 Админ-панель (Filament v5)

| Ресурс | Возможности |
|--------|-------------|
| **Товары** | CRUD, изображения, rich-editor описание, фильтры, массовая активация/деактивация |
| **Категории** | CRUD с иерархией, иконки, порядок сортировки |
| **Водители** | CRUD аккаунтов, управление стоком через RelationManager |
| **Заказы** | Просмотр, смена статуса, **фильтр по типу заказа** (на месте / доставка), фильтры по статусу/оплате/водителю/дате |
| **Дашборд** | Виджеты: статистика, график продаж, топ-товары, последние заказы |

### 2.10 Мультиязычность

| Язык | Код | Статус |

## 4. Бизнес-логика: основные сценарии

### Сценарий 1: Покупка на месте (Pickup)
1. Планшет авторизован под аккаунтом водителя
2. Пассажир просматривает каталог на планшете (заднее сиденье)
3. Видит товары, которые есть в машине (помечены бейджем)
4. Нажимает «Купить на месте» → попадает на checkout
5. Выбирает способ оплаты (наличные / QR / при получении), имя/телефон НЕ нужны
6. Заказ создаётся со статусом `pending`
7. На экране показывается «Ожидание подтверждения водителя»
8. Водитель на телефоне видит новый заказ в `/driver/orders`
9. Водитель нажимает «Подтвердить» → сток списывается, товар передаётся клиенту

### Сценарий 2: Доставка на дом (Delivery)
1. Пассажир видит товар, которого нет в машине, но доступен для доставки
2. Нажимает «Заказать доставку» → попадает на checkout с формой
3. Вводит имя, телефон (+998), адрес доставки, выбирает способ оплаты
4. Заказ создаётся со статусом `pending`
5. Водитель подтверждает заказ → статус `confirmed`, сток НЕ списывается
6. Доставка обрабатывается отдельно

### Сценарий 3: Управление каталогом (админ)
1. Админ входит в `/admin`
2. Создаёт/редактирует категории и товары
3. Назначает товары водителям через DriverResource → StockRelationManager
4. Отслеживает заказы с фильтром по типу (на месте / доставка)
5. Может менять статусы заказов

---

## 5. Статусы заказов

| Статус | Описание |
|--------|----------|
| `pending` | Ожидает подтверждения водителем |
| `confirmed` | Подтверждён водителем (pickup: товар передан; delivery: заказ принят) |
| `delivered` | Доставлен (для delivery-заказов) |
| `cancelled` | Отменён/отклонён |

---

## 6. Ограничения и валидация

- Телефон: формат `+998XXXXXXXXX` (Узбекистан, только для delivery)
- Slug: уникальный для товаров и категорий
- Номер заказа: уникальный, формат `TS-######`
- Удаление товара: запрещено, если есть связанные заказы (restrict)
- Удаление пользователя: запрещено, если есть связанные заказы (restrict)
- Каскадное удаление: driver_stock при удалении водителя/товара, order_items при удалении заказа
- Pickup-заказы: customer_name и customer_phone могут быть null
- Delivery-заказы: customer_name, customer_phone и delivery_address обязательны
